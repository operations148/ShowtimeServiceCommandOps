# Change Order & Work-Order Project Test Plan — Phase 5

Maps the Phase-5 required test list to concrete verification. Unit-tested items are automated (Vitest); the rest are manual steps to run against a deployed preview with the migration applied (several depend on a real DB / real browser / production-like config). Run before Phase 5 is considered closed.

| # | Test | Automated? | Verification |
|---|---|---|---|
| 1 | Work-order state machine | Yes (`state-machine.test.ts`, 15 tests) | Illegal transitions rejected; `ARCHIVED` is terminal; `CLOSED`/`ARCHIVED` unreachable except via dedicated actions. |
| 2 | Change-order state machine | Yes (`state-machine.test.ts`, 12 tests) | `isEditable` only true for `draft`; `isPending` = draft/sent/viewed; `voided` terminal. |
| 3 | Change-order totals | Yes (`totals.test.ts`, 7 tests) | Price/tax/total impact computed from all lines (no selection logic); cost impact summed separately. |
| 4 | Completion-requirements evaluator | Yes (`completion-requirements.test.ts`, 10 tests) | Each configured-required field, when missing, blocks completion; all-satisfied passes. |
| 5 | Public change-order serializer redaction | Yes (`public-serializer.test.ts`, 9 tests) | Serialized JSON contains no `cost_impact_cents`, `unit_cost`, `tenant_id`, staff ids, or internal notes. |
| 6 | Cost redaction (admin API) | Yes (`redact-costs.test.ts`) | As OFFICE_STAFF, GET `/api/change-orders/[id]` → no `unit_cost`/`cost_impact_cents` on the response. |
| 7 | Shared token/PDF-text utilities | Yes (`public-document-token.test.ts`, `pdf-text.test.ts`, 16 tests combined) | 256-bit token, SHA-256 hash-at-rest, expiry/revocation logic; control chars stripped and length-capped for PDF text. |
| 8 | Permission matrix | Yes (`roles.test.ts`, Phase 5 block) | `canCloseWorkOrders`/`canManageWorkOrderTasks`/`canManageWorkOrderAttachments`/`canManageChecklistTemplates`/`canManageCompletionRequirements`/`canViewChangeOrders`/`canManageChangeOrders`/`canVoidChangeOrders`/`canOverrideChangeOrderLock`/`canApplyScheduleImpact` pinned per role. |
| 9 | Parent/child authorization | No | Create a child WO under Tenant A's project; confirm a Tenant B session gets 404 on both the parent and the child via `/api/work-orders/[id]`. |
| 10 | Multi-visit project | No | Create a parent WO, add 2 children via `POST /children`; confirm parent's `is_multi_day` flips to `true` and both children appear under `GET /children`; each child schedules its own visit/technician independently. |
| 11 | Required-completion-fields integration | No | Configure a `completion_requirement_rules` row requiring `customer_signature`; PATCH a visit to `completed` without one → 422 listing the missing field; no DB write occurs (confirm visit `status` unchanged). Then supply the signature → 200, and confirm a `visit_checklist_snapshots` row was written. |
| 12 | Checklist-snapshot immutability | No | Complete a visit, then edit the checklist template's items; confirm the earlier `visit_checklist_snapshots` row for that visit is unchanged. |
| 13 | Archive instead of delete | No | `DELETE /api/work-orders/[id]` on a closed WO; confirm the row still exists in the DB with `archived_at` set (not removed), and `GET` on the id via a direct DB query still returns the row. |
| 14 | Change-order totals via API | No | Create a change order with 2 line items + tax; confirm `price_impact_cents`/`tax_impact_cents`/`total_impact_cents` on the GET response match server-computed values, ignoring any client-sent totals. |
| 15 | Public token resolution + expiry | No | Send a change order; GET the public link; set `token_expires_at` in the past; GET again → generic 404. |
| 16 | Accept/reject replay (idempotency) | No | POST accept twice with the same token → second returns `{ alreadyDecided: true }`; confirm the parent WO's `approved_contract_amount_cents` was only bumped once (not twice). |
| 17 | Pending-change-order invoicing/closeout block | No | Create a change order with `blocks_closeout: true` on a `completed` WO, leave it in `sent`; `POST /close` → 409 with `changeOrderIds` containing it. Accept or void it; `POST /close` → 200. |
| 18 | Contract-value update on accept | No | Note the WO's `approved_contract_amount_cents`; accept a change order with `price_impact_cents = 5000`; confirm the WO's value increased by exactly 5000, and a `contract_value_applied` event was recorded. |
| 19 | Reject never alters contract value | No | Reject a change order; confirm the parent WO's `approved_contract_amount_cents` is unchanged. |
| 20 | Override permission + reason | No | As OFFICE_STAFF, POST `/override` on an accepted change order → 403 (`canOverrideChangeOrderLock: false`). As TENANT_ADMIN without a reason → 422. With a reason → re-opens to `draft`, revokes the prior token, does **not** reverse the already-applied contract value (confirm WO's `approved_contract_amount_cents` unchanged from #18). |
| 21 | Apply-schedule-impact explicitness | No | Accept a change order with `schedule_impact_days: 3`; confirm no visit was rescheduled automatically. Call `POST /apply-schedule-impact` with a visit id + new date → visit's `scheduled_date` updates and `schedule_impact_applied_at` is set; calling it again → 409 "already applied". |
| 22 | Cross-tenant denial | No | A Tenant B session requesting Tenant A's change order (`GET /api/change-orders/[id]`) → 404, not 403 (no existence oracle). Same for the public token — a token minted for Tenant A never resolves under any Tenant B context (there is none; tenant is derived from the resolved row, never the caller). |
| 23 | PDF escaping (completion report) | Yes (`pdf-text.test.ts`) + manual | Control chars stripped by the shared `pdfText()` sanitizer, applied to every interpolated value in the upgraded `/api/work-orders/[id]/report` route (title, description, property fields, visit notes, checklist labels, materials, signature, change-order reason/number). Manually: set a visit's `customer_signature` to a hostile string with control characters; download the report; confirm no corruption. |
| 24 | Completion-report permission enforcement | No | As TECHNICIAN, GET `/api/work-orders/[id]/report` → 403 (`canViewAllWorkOrders: false`). As OFFICE_STAFF (no `canViewFinancialReports`) → 200, but the Contract Value section and change-order price impacts are omitted from the PDF. As TENANT_ADMIN → 200 with financial figures included. |
| 25 | Rate limiting (public routes) | No | Hit `GET /api/public/change-orders/[token]` >30×/min from one IP → 429; decisions (`accept`/`decline`) >10/hr → 429 — same policies as public estimates. |
| 26 | File-upload validation (attachments) | No | Upload a `.exe` renamed to `.pdf` to `/api/work-orders/[id]/attachments` → rejected by the magic-byte sniff (422), not accepted based on filename/extension alone. |
| 27 | Technician task-completion scoping | No | As TECHNICIAN, PATCH `/api/work-orders/[id]/tasks/[taskId]` for a task NOT assigned to them → 404. For their own task, `{ is_completed: true }` → 200; `{ title: "new" }` on their own task → 403 ("Technicians may only update completion status"). |

## Sign-off

Run the manual rows against a deployed preview (not local dev — rate limits, headers, storage, and email modes depend on production-like config) per `qa/launch-readiness-checklist.md`'s "no phase ships without QA sign-off" rule.
