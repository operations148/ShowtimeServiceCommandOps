# Security Audit ‚Äî Phase 0

_Compiled 2026-07-11 from a full-repository grep/read sweep (three independent passes: manual review of auth/RLS/webhook/delete code, a route-by-route API audit, and a targeted grep sweep) plus first-hand verification of every finding below against the actual source. Severity rubric: **Critical** = direct cross-tenant data exposure or auth bypass with no precondition; **High** = exploitable but requires a specific (plausible) condition, or a broken/absent security control; **Medium** = missing defense-in-depth or a reliability risk with security implications; **Low** = hygiene/best-practice gap with low exploit likelihood._

**Headline: zero Critical findings** (no unconditional cross-tenant data exposure was found ‚Äî the service-role-client + `getTenantId()` pattern is applied consistently across all 35 API routes). However, several **High** findings mean the app's security posture is weaker than its own documentation claims, and the two-invoice-model / two-untracked-tables schema drift is a real correctness risk for the phases ahead.

## Critical

None found.

## High

| # | Finding | Evidence | Why it matters |
|---|---|---|---|
| H1 | **No login rate limiting, lockout, or throttling anywhere.** | `src/lib/auth/config.ts` `authorize()` ‚Äî full read confirms no attempt counter, no delay, no CAPTCHA. Repo-wide grep for `lockout|attempts|throttle|rate.?limit` returns zero auth-related hits. | Credential-stuffing / brute-force against `/api/auth/callback/credentials` is mitigated only by bcrypt cost-12, not by any request-level control. |
| H2 | **No session revocation mechanism.** Deactivating a user (`is_active: false`) or changing their role does not invalidate an already-issued JWT ‚Äî it remains valid for up to its full 8-hour `maxAge`. | `src/lib/auth/config.ts:55-71` ‚Äî `jwt()` callback only sets claims at sign-in and never re-validates against the DB; `is_active` isn't even carried into the token. No `session_version` column exists anywhere (`grep session_version` ‚Üí no matches). | An admin who deactivates a compromised or terminated user's account, expecting immediate effect, has not actually cut off that user's access for up to 8 hours. This is a broken security control, not just a missing one. |
| H3 | **`GET /api/cron/generate-visits` fails open if `CRON_SECRET` is unset.** | `const secret = process.env.CRON_SECRET; if (secret && authHeader !== ...) return 401;` ‚Äî if `secret` is falsy, the whole condition short-circuits and the check never runs. | This route iterates every tenant and mutates work orders/visits. An unset env var in production silently makes it public, with no build-time or boot-time signal. Contrast with the GHL webhook route, which correctly fails closed (`503`) in production when its secret is unset. |
| H4 | **`POST /api/work-orders/[id]/send-estimate` has no permission gate or ownership check ‚Äî any authenticated role, including an unrelated TECHNICIAN, can send an arbitrary attacker-controlled `recipientEmail` a message containing the property's gate code, access notes, customer name/address, and technician name.** | Route uses bare `requireApiAuth()` only (no `requirePermission`, no work-order/technician ownership check), while every other mutating work-order route uses `requirePermission(...)`. | Direct PII/physical-security-relevant data exfiltration primitive (gate codes!) to an arbitrary destination, plus an unrate-limited email-sending primitive against any target inbox using the app's own Resend reputation. |

## Medium

| # | Finding | Evidence |
|---|---|---|
| M1 | **RLS is enabled on every tenant table but is currently inert for all application traffic.** All policies gate on `current_tenant_id()`/`current_user_id()`/`current_user_role()`, which resolve from `auth.jwt()` (never populated ‚Äî the app uses NextAuth, not Supabase Auth) or `current_setting('app.current_tenant_id', ...)` (never set anywhere in the TypeScript codebase). Every query goes through the service-role client (`src/lib/db/client.ts`), which bypasses RLS entirely. `src/lib/db/browser.ts` (anon-key client, the only path that would make RLS matter) has zero call sites. | Not exploitable today (nothing uses the anon-key path), but it means tenant isolation has exactly one enforcement layer ‚Äî `getTenantId(session)` + manual `.eq("tenant_id", ...)` in every query ‚Äî with no working backstop if a future route forgets it. |
| M2 | **`deleteWorkOrder()` and `deleteRecurringSchedule()` both accept an optional, unenforced `tenantId?: string` parameter** rather than requiring it. | `src/lib/db/queries/work-orders.ts:458-463`, `src/lib/db/queries/recurring-schedules.ts:176-185`. Both current call sites do pass `tenantId`, so this is not exploited today ‚Äî it's a landmine for the next caller (script, cron, refactor) that omits the argument. |
| M3 | **`deleteWorkOrder` is a real hard delete with a large cascade blast radius**, irreversibly destroying `visits`, `estimate_handoffs`, and the entire `work_order_status_history` audit trail (`ON DELETE CASCADE` on all three), with no soft-delete alternative ‚Äî `work_orders` has no `is_active`/`archived_at` column at all, only a `status` enum that includes `cancelled` but isn't used instead of delete. `recurring_schedules` has an `is_active` column that exists but is ignored by its own delete route (hard-deletes instead). | `supabase/migrations/20260506000005/006/010`, `20260514000002`; `src/app/api/work-orders/[id]/route.ts` DELETE handler; `src/app/api/recurring-schedules/[id]/route.ts` DELETE handler. |
| M4 | **No file-content validation (magic-byte sniffing) anywhere** ‚Äî avatar, company-logo, and job-photo uploads all trust only the client-supplied `Blob.type` header, which is fully spoofable. No `file-type`/similar library is installed. | `src/lib/storage/avatars.ts`, `src/lib/storage/photos.ts`, `src/app/api/settings/company/logo/route.ts`. |
| M5 | **Company logo upload whitelists `image/svg+xml` into a public storage bucket, with no CSP configured anywhere and no SVG sanitizer applied.** | `src/app/api/settings/company/logo/route.ts` `ALLOWED_TYPES` includes `image/svg+xml`. Currently rendered only via `<img src>` in the one place found (`SettingsPageClient.tsx`), which doesn't execute embedded scripts, but the object is also directly URL-addressable outside that context. |
| M6 | **No EXIF/GPS metadata stripping on any uploaded image** (avatars, logos, job photos) ‚Äî a technician's phone photo retains embedded GPS coordinates in the stored file. | Confirmed by full read of `src/lib/storage/photos.ts` ‚Äî no image-processing/EXIF-strip step exists. |
| M7 | **Cross-visit photo deletion (same-tenant IDOR).** `DELETE /api/visits/[id]/photos` verifies the caller owns the visit named in the URL, and verifies the `path` in the request body starts with the tenant's storage prefix ‚Äî but never verifies the `path` belongs to *that specific visit's* own `photo_urls`. Any user who owns at least one visit in the tenant can supply another visit's photo path and delete it. | `src/app/api/visits/[id]/photos/route.ts` DELETE handler. |
| M8 | **`POST /api/visits` performs no ownership verification** that the supplied `work_order_id`/`property_id` belong to the caller's tenant, nor that a caller-supplied `technician_id` is a technician within that tenant ‚Äî and has no permission gate beyond `requireApiAuth()` (any role, including TECHNICIAN, may call it), unlike the GET handler which correctly force-scopes technicians. | `CreateVisitSchema` (`src/lib/validation/visit.ts`) only requires non-empty strings; `src/app/api/visits/route.ts` POST handler. |
| M9 | **No security headers configured anywhere** ‚Äî no CSP, HSTS, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, or `frame-ancestors`. | Full read of `next.config.ts` ‚Äî no `headers()` function defined at all. |
| M10 | **No CSRF-token layer for any API route**; protection is entirely implicit via NextAuth's default `SameSite=Lax` cookie. The three multipart-form-data upload routes (avatar, logo, photos) don't trigger a CORS preflight and could in principle be submitted via an auto-submitting cross-site form ‚Äî the only defense is the browser's default SameSite behavior, with zero server-side backstop. | Repo-wide grep for `Origin|Referer|csrf` under `src/app/api` ‚Üí zero hits; `src/lib/auth/config.ts` has no custom `cookies` block. |
| M11 | **`user_invitations` table has no tracked migration anywhere** ‚Äî created directly against the live Supabase project, like `invoices`. Its token column stores the invite token **in plaintext** (not hashed, unlike `password_hash`), looked up by direct equality. | `src/app/api/invitations/accept/route.ts` ‚Äî `.eq('token', token)`; repo-wide grep of `supabase/migrations/` for `user_invitations` ‚Üí zero hits. |
| M12 | **Invitation acceptance is not atomic ‚Äî a TOCTOU race allows double-redemption of the same token.** The handler does a plain SELECT for `accepted_at`, then two independent, unconditional UPDATEs with no `WHERE accepted_at IS NULL` guard and no unique-constraint race-catch (contrast with `create-invoice-from-estimate.ts`, which does handle a `23505` race correctly). | `src/app/api/invitations/accept/route.ts`. |
| M13 | **`invoices.public_token`'s generation mechanism is unauditable** ‚Äî the base `invoices` table (created via the Supabase dashboard, not a migration) presumably defaults this column via `gen_random_uuid()`, but this cannot be confirmed from source. It is already live in a customer-facing redirect URL (`src/lib/stripe/deposit-checkout.ts` constructs `/estimate/${invoice.public_token}?status=paid`) pointing at a page that does not exist (see repository-inventory.md module status). | `supabase/migrations/20260613000001_invoices.sql` header comment; `src/lib/stripe/deposit-checkout.ts`. |
| M14 | **GHL webhook route logs secret-adjacent metadata** ‚Äî on every mismatched auth attempt it logs the exact byte-length of `GHL_WEBHOOK_SECRET` and whether the caller's guessed token matches its first character. | `src/app/api/ghl/webhooks/route.ts`: `reason: \`bearer_mismatch | headerTokenLen=${tokenFromHeader.length} secretLen=${secretTrimmed.length} firstCharMatch=${tokenFromHeader[0] === secretTrimmed[0]}\`` ‚Äî then `console.log`'d. |
| M15 | **Raw customer email is logged on every failed login attempt**, and customer full names are logged repeatedly in GHL-sync code paths, contradicting the project's own "PII handled with care" rule. | `src/lib/auth/config.ts:26,31,37`; `src/lib/ghl/upsert-property-from-ghl.ts`, `work-order-factory.ts`. |
| M16 | **`scripts/validate-env.sh` exists but is never invoked** by `npm run dev/build/start`, by `next.config.ts`, or by any CI workflow (none exists) ‚Äî the ".env validated at startup" rule in `CLAUDE.md`/`security-rules.md` is aspirational documentation, not wired-up behavior. Its required-variable list is also stale (checks unused `NEXT_PUBLIC_APP_NAME`/`DATABASE_URL`; omits load-bearing vars like `SUPABASE_SERVICE_ROLE_KEY`, `STRIPE_SECRET_KEY`, `CRON_SECRET`). | `package.json` scripts block; full read of `scripts/validate-env.sh`. |
| M17 | **Two incompatible `InvoiceStatus` type systems coexist** ‚Äî `src/types/estimate.ts` (fully unused/dead, zero imports found) defines one `InvoiceStatus`/`Estimate`/`Invoice` model referencing tables (`estimates`, `invoice_items`, `payments`) that don't exist; `src/types/invoice.ts` (the real, live one) defines an incompatible `InvoiceStatus` enum matching the actual DB enum. `estimate_handoffs` is a third, independent, actually-used state machine matching neither. | `src/types/estimate.ts` vs `src/types/invoice.ts` vs `supabase/migrations/20260506000010_create_estimate_handoffs.sql`. |
| M18 | **`users.email` is unique only per-tenant (`UNIQUE(tenant_id, email)`), but the login `authorize()` callback queries by email alone with no tenant/slug disambiguation.** If the same email is ever active in two tenants, Supabase's `.single()` will error for both (login breaks for that email in both tenants, a correctness/availability bug) rather than silently selecting the wrong tenant ‚Äî so this is not a cross-tenant data leak, but it is exactly the "same email across multiple organizations" identity-model gap the next phase of work needs to resolve deliberately, per its own explicit instruction not to leave this ambiguous. | `supabase/migrations/20260506000003_create_users.sql:19`; `src/lib/auth/config.ts:18-23`. |

## Systemic gaps (not single-finding, apply across the whole app)

- **No rate limiting anywhere** ‚Äî login, invitation acceptance, file uploads, reports, webhooks, cron. No rate-limiting dependency exists in `package.json`.
- **`src/middleware.ts` does not cover `/api/*` at all** ‚Äî there is no shared authentication/rate-limit/origin gate in front of any API route; each of the 35 route files is independently responsible for its own checks, and this audit found the checks are *almost* always present and correct, with H4 (`send-estimate`) as the one clear gap.
- **No structured logger, no custom error boundaries** (`error.tsx`/`global-error.tsx` don't exist) ‚Äî all logging is ad hoc `console.*`, contradicting the project's own coding-standards rule.
- **Two database tables (`invoices`, `user_invitations`) exist with no tracked migration** ‚Äî created via the Supabase dashboard, invisible to `supabase/migrations/`, and (per the `invoices` history) prone to the same missing-grants bug unless independently checked. `invoice_line_items` is referenced in code but has no migration and its existence/grants are unverified ‚Äî line-item snapshots on invoices may be silently failing today.

## What this audit did *not* verify

- Whether the `properties`/`avatars` Storage buckets are configured public or private, and whether any bucket-level Storage RLS policies exist ‚Äî this is configured outside version control and needs direct confirmation in the Supabase dashboard.
- The exact column list of the untracked `invoices`/`user_invitations`/`invoice_line_items` tables ‚Äî reconstructed from TypeScript row types, not from a live `information_schema` dump.
- Whether `invoice_line_items` exists at all, or whether its inserts are currently failing silently (the code catches and only logs the error).
